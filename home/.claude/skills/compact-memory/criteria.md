# メモリ判定基準

各メモリファイルを以下の基準で「保持 / 削除 / 書き換え / 統合」に分類する。

## 大原則

- git・コード・リポジトリから辿れる情報はメモリに持たない
- 削除は実際の検証で裏付けが取れた場合のみ行う。推測で削除しない
- 迷ったら保持し、ユーザーに質問する

## 削除

以下のいずれかに該当し、かつ検証で確認できたもの。

| 対象 | 検証方法 |
| --- | --- |
| コード構造・設定値・ツール構成など、リポジトリを読めば分かる情報 | 該当ファイルを Grep / Read で実在確認 |
| 過去の修正内容・経緯など、git 履歴に記録済みの情報 | `git log` で該当コミットを確認 |
| 完了済み PR・マージ済み計画などの作業履歴 | `gh pr view` / `git log` で完了を確認 |
| CLAUDE.md に既に記載されている指示・ルール | CLAUDE.md を Read して確認 |
| 他メモリまたは現状と矛盾する記述 | 現状と一致する側を確認して残す。正誤不明ならユーザーに質問 |

## 書き換え（縮約）

- 完了済み作業の記録だが、普遍的な教訓・ルールを含むもの → 教訓部分のみ残して縮約
- 古くなった詳細を含むが、核となる事実は有効なもの → 有効な部分のみ残す

## 統合

- 同一トピックに分散した複数メモリ → 1 ファイルに統合し、`[[リンク]]` を貼り替える

## 保持

- `metadata.type: user` / `feedback` — ユーザーの特性・作業ルールは原則保持
- リポジトリから導出できない運用知識（外部システムの挙動、障害対応の教訓など）
- `metadata.type: reference` のうち、参照先がまだ有効な外部参照
