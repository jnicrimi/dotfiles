# compact-memory

## 概要

現在のプロジェクトの永続メモリを分析し、不要になったメモリの削除・統合と MEMORY.md インデックスの更新を行う。削除は不可逆のため、必ず「提案 → 承認 → 適用」の順で進める。

## 判定基準

[criteria.md](./criteria.md) を参照

## 実行手順

### 1. メモリの読み込み

- メモリディレクトリはシステムプロンプトに記載された永続メモリのパスを使用する。shell でパスを再構築しない
- `MEMORY.md` と全メモリファイルを読み込む
- メモリファイルが存在しない場合は「コンパクト化対象のメモリがありません。」と出力して処理を終了

### 2. 各メモリの判定

[criteria.md](./criteria.md) の基準に従い、各メモリファイルを「保持 / 削除 / 書き換え / 統合」に分類する。

- **MUST**: 「リポジトリから辿れるはず」という推測だけで削除判定しない。grep・`git log`・`gh pr view` 等で実際に確認できた場合のみ削除候補とする
- **MUST**: 判定を確定できないもの（矛盾の正誤が判断できない等）は保持とし、提案時にユーザーへ質問する

### 3. 提案の提示

判定結果を以下の形式で提示し、ユーザーの承認を待つ。

```text
🗜️ メモリコンパクト化の提案
────────────────────────────────────────
対象: [メモリディレクトリのパス]（[N] ファイル / [サイズ]）

| ファイル | 判定 | 理由 |
|---------|------|------|
| ...     | 保持 / 削除 / 書き換え / 統合 | [検証結果を含む理由] |

[書き換え・統合がある場合は変更後の内容案を提示]

このまま適用しますか？（個別に判定を変更できます）
```

- 承認された場合は次へ進む
- 個別の変更指示があった場合は判定を修正して再提示する

### 4. 適用

承認された判定に従って適用する。

- 削除: `rm` でファイルを削除
- 書き換え・統合: Edit / Write で更新。frontmatter（`name` / `description` / `metadata.type`）は維持する
- 削除・統合により宙ぶらりんになった `[[リンク]]` を修正する

### 5. MEMORY.md インデックスの更新

残存するメモリファイルと 1:1 で一致するよう MEMORY.md を再生成する。

- 形式: `- [Title](file.md) — 概要`（1 メモリ 1 行）
- メモリの内容そのものはインデックスに書かない

### 6. 完了報告

以下を報告する。

- 削除 / 書き換え / 統合 / 保持の件数
- ビフォーアフターのファイル数とサイズ
